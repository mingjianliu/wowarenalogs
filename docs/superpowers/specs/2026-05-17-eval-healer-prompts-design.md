# Healer Prompt/Response Evaluation Pipeline — Design Spec

**Date:** 2026-05-17
**Status:** Approved

## Goals

Two equally-weighted goals:

1. **Improvement compass** — systematically score prompts and responses to identify the highest-leverage areas to fix (e.g., "Disc Priest loss prompts consistently score low on inference scaffolding")
2. **Regression guard** — detect when a code change to `buildMatchPromptNew` or the analysis utilities makes prompts or responses worse than a saved baseline

Human acts as the final decision-maker on all changes. The pipeline flags issues; humans decide what to fix.

## Skill

**Name:** `eval-healer-prompts`

**Modes:**

| Invocation                             | Behavior                                                     |
| -------------------------------------- | ------------------------------------------------------------ |
| `/eval-healer-prompts`                 | Fetch fresh matches, run full eval, produce report           |
| `/eval-healer-prompts --snapshot`      | Use saved snapshot corpus (regression guard mode)            |
| `/eval-healer-prompts --save-snapshot` | Save current `prompts/` directory as the regression baseline |

## Pipeline — Four Steps

### Step 1: Build Corpus

Run `buildHealerPromptCorpus.ts` to fetch real 3v3 matches from the production GraphQL `latestMatches` feed. Finds matches where the log owner is a healer spec and writes one structured prompt file per match.

- **Default mode**: always fetches fresh matches
- **Snapshot mode (`--snapshot`)**: skips this step, uses `prompts-snapshot/` as the corpus

Target: 10–50 matches per run. Spec quota enforced (≤ `ceil(target / 7)` per healer spec) to avoid over-representing one spec.

### Step 2: Generate Responses (Parallel Sub-Agents)

Spawn N background sub-agents in parallel — one per prompt file. Each sub-agent:

1. Reads `prompts/NNN.txt`
2. Acts as an arena coach and generates a coaching response
3. Writes the response to `responses/NNN.txt`

Sub-agents run with `run_in_background: true`. The main agent receives completion notifications automatically and proceeds to Step 3 once all `responses/NNN.txt` files exist (verified by checking against `index.json` entry count).

**Sub-agent instruction template:**

> You are a WoW arena coach. Read the match prompt below and produce coaching advice for the healer. Write your response to `packages/tools/local-batch/healer-eval/responses/NNN.txt`. Coaching only — no meta-commentary about the prompt or your reasoning process.

This separation ensures the model generating responses and the model evaluating them operate in different conversation contexts, reducing self-evaluation bias.

### Step 3: Score Each Match

Main agent reads each `prompts/NNN.txt` + `responses/NNN.txt` pair in sequence and scores on the rubric below. Writes `scores/NNN.json` after each pair.

No additional API calls — the main agent (this Claude Code session) performs evaluation inline.

### Step 4: Synthesize Report

Main agent reads all `scores/*.json` and writes `eval-report.md` containing:

- **Aggregate score distribution** — histogram per dimension across all matches
- **Flagged matches** — any match with any dimension scoring ≤ 2, listed with the failing dimension and score
- **Cross-spec patterns** — average scores broken down by healer spec (e.g., "Disc Priest: sufficiency avg 2.4, label bias avg 1.8")
- **Top 3 issues** — ranked by frequency × severity, with concrete descriptions of what to fix

## Evaluation Rubric

Scores are integers 1–5. Each score includes a one-sentence justification. All scores are written to `scores/NNN.json`.

### Prompt Quality

| Dimension                       | What it measures                                                                                                                                                      |
| ------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Sufficiency** (1–5)           | Does the prompt contain enough data for Claude to identify what actually mattered? Missing CC chain, no dampening value, no enemy CD timeline → low score             |
| **Noise** (1–5)                 | Are there verbose or redundant sections unlikely to contribute to useful inference? Repeated unchanged `[RES] rdy:` lines, passive proc spam → low score              |
| **Label bias** (1–5)            | Do section labels or framing steer Claude toward a conclusion before it reasons? Mislabeled severity, loaded language in headers → low score                          |
| **Inference scaffolding** (1–5) | Are events ordered and labeled so Claude can connect cause → effect? Death appearing near the damage that caused it, CC timing relative to trinket usage → high score |

### Response Quality

| Dimension                   | What it measures                                                                                                                                 |
| --------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Accuracy** (1–5)          | Does the response reference only events present in the prompt? Hallucinated spell names, wrong timestamps, invented enemies → low score          |
| **Outcome alignment** (1–5) | Does the response explain factors that plausibly contributed to the win or loss? Ignoring a 3-minute loss with no defensive CDs used → low score |
| **Focus calibration** (1–5) | Does Claude identify the highest-leverage moments, or treat all events as equal? Surface-level recap with no prioritization → low score          |

### Per-Match Score Schema

```json
{
  "matchId": "abc123",
  "spec": "Priest_Discipline",
  "result": "Loss",
  "durationSec": 187,
  "prompt": {
    "sufficiency": 3,
    "noise": 4,
    "labelBias": 2,
    "inferenceScaffolding": 3,
    "notes": "CC chain section present but no trinket timing context. Label bias: [CRITICAL] applied to a minor HP dip."
  },
  "response": {
    "accuracy": 5,
    "outcomeAlignment": 2,
    "focusCalibration": 3,
    "notes": "Response does not mention the two consecutive Fear chains in the final minute that directly preceded the loss."
  }
}
```

## Feedback Signal

| Signal                               | Role                                                                                                                        |
| ------------------------------------ | --------------------------------------------------------------------------------------------------------------------------- |
| Win/loss outcome (from `index.json`) | Primary anchor for outcome alignment scoring — a response that fails to explain a loss is a concrete, measurable failure    |
| AI judge (main agent, Step 3)        | Applies rubric using prompt as factual ground truth; checks for internal consistency                                        |
| Human spot-checks (occasional)       | After runs reach ~80% scoring confidence, human reviews flagged matches to validate AI judge quality and recalibrate rubric |

## Snapshot / Regression Guard

The `--save-snapshot` command copies `prompts/` to `prompts-snapshot/`. Subsequent `--snapshot` runs use `prompts-snapshot/` as the corpus, holding inputs constant. This allows before/after comparison of code changes to `buildMatchPromptNew` or analysis utilities without confounding from different match batches.

Snapshots are not committed to git (too large). They live on disk per developer.

- Spec-specific rubric adjustments (e.g., Disc vs. Holy Priest coaching criteria)
- Committing eval reports or snapshots to git
